# 性能評価

このページでは、LSQSolver と GNU Octave の予備的なベンチマーク結果を示します。

ベンチマークでは、以下のサイズの密な正方行列を使用しました。

`n = 10, 50, 100, 500, 1000, 2000`

また、次の2種類のランク条件を用いました。

* `rank_half`: 数値ランクが `n / 2` の正方行列
* `full_rank`: 数値ランクが `n` の正方行列

各値は10回実行したときの中央値です。
時間の単位はすべてミリ秒です。

## ベンチマーク環境

ベンチマークは以下の環境で実行しました。

| 項目     | 値                        |
| ------ | ------------------------ |
| 機種名    | MacBook Pro              |
| チップ    | Apple M1 Pro             |
| CPUコア数 | 合計8コア: パフォーマンスコア6、効率性コア2 |
| メモリ    | 16 GB                    |

## Octave QR 分解との比較

LSQSolver は列ピボット付き QR 分解を基礎としているため、Octave 側で最も近い比較対象は `qr_factor` です。

| case / solver                |       10 |     50 |    100 |      500 |     1000 |      2000 |
| ---------------------------- | -------: | -----: | -----: | -------: | -------: | --------: |
| rank_half / LSQSolver        | 11.2097  | 3.5472 | 6.9870 |  96.9781 | 338.5879 | 1353.3620 |
| rank_half / Octave qr_factor |  0.1     | 0.6    | 2.7    |  80.9    | 483.2    | 2102.2    |
| full_rank / LSQSolver        |  0.08455 | 0.4121 | 1.0912 |  23.0219 | 148.3778 | 1171.7226 |
| full_rank / Octave qr_factor |  0.0     | 0.5    | 2.2    |  67.8    | 258.1    | 1482.95   |

## 考察

非常に小さい行列では、計算そのものよりも関数呼び出しや初期化などのオーバーヘッドの影響が大きくなります。

フルランク行列では、このベンチマークにおいて LSQSolver は `n >= 50` で Octave の QR 分解より高速でした。

ランク落ち行列では、小さいサイズでは LSQSolver は Octave の QR 分解より遅い一方で、大きいサイズでは Octave の QR 分解より高速でした。このベンチマークでは、`n = 1000` および `n = 2000` において LSQSolver は Octave の QR 分解より高速でした。

`n = 2000` のランク落ちケースでは、LSQSolver が `1353.4 ms`、Octave の QR 分解が `2102.2 ms` でした。

Octave の `pinv` と比較すると、大きな行列に対して LSQSolver は大幅に高速です。`n = 2000` では、ランク落ちケースで約 `7.2倍`、フルランクケースで約 `13.3倍` 高速でした。

## 注意

これらの結果は予備的なものであり、ハードウェア、実行環境、コンパイラ設定、BLAS/LAPACK の構成、およびベンチマーク実装に依存します。

Octave の `qr_factor` の時間は QR 分解の計算時間を測定したものであり、LSQSolver がランク落ちケースで行っている最小ノルム解の構成全体を必ずしも含むものではありません。そのため、この比較は厳密なアルゴリズム単位の比較ではなく、実用上の参考値として解釈してください。
